## Можливості

| Функція | Деталі |
|---------|--------|
| NL → SQL | Описуйте запит словами — отримуйте готовий SQL |
| Schema-aware | Модель бачить реальну схему БД і не вигадує неіснуючих таблиць/стовпців |
| Structured output | Відповідь валідується Pydantic (`SQLQueryResult`) |
| Few-shot prompting | 2 вбудовані приклади направляють модель до стабільного, якісного виводу |
| Локальний кеш | Повторні ідентичні запити повертаються з диска (TTL = 1 год.) |
| Warnings | Потенційні проблеми з продуктивністю виявляються автоматично |
| Альтернативи | Опціонально — альтернативні SQL-підходи |
| Download | Скачати `.sql`-файл в один клік |
| Streamlit UI | Зручний веб-інтерфейс з 3 вбудованими схемами БД |

## Швидкий старт

### 1. Передумови

```bash
# Встановити uv (якщо ще не встановлено)
# macOS / Linux:
curl -LsSf https://astral.sh/uv/install.sh | sh

# Windows (PowerShell):
powershell -c "irm https://astral.sh/uv/install.ps1 | iex"
```

### 2. Клонувати та налаштувати

```bash
git clone <repo-url>
cd smart-sql-architect

# Створити .venv та встановити залежності одною командою
uv sync
```

### 3. Налаштувати API-ключ

```bash
cp .env.example .env
# Відкрийте .env і вставте свій ключ:
# GEMINI_API_KEY=your_key_here
```

> Безкоштовний ключ: https://aistudio.google.com/app/apikey

### 4. Запустити застосунок

```bash
uv run streamlit run src/sql_architect/main.py
```

Браузер відкриється автоматично за адресою `http://localhost:8501`.

### 5. (Опціонально) Запустити тести

```bash
uv run pytest -v
```

## Як це працює

```
Запит користувача
       │
       ▼
Серіалізація схеми БД → текстовий формат
       │
       ▼
Few-shot промпт (2 приклади) + запит + схема
       │
       ▼
Gemini 1.5 Flash  (system_instruction визначає роль)
 temperature=0.1 · top_p=0.85 · top_k=20
       │
       ▼
JSON-відповідь → Pydantic-валідація → SQLQueryResult
       │
       ▼
Streamlit UI: SQL, пояснення, попередження, download
```

## Параметри моделі

| Параметр | Значення | Обґрунтування |
|----------|----------|---------------|
| `temperature` | **0.1** | SQL має бути коректним, а не творчим — майже детермінований вивід |
| `top_p` | **0.85** | Трохи обмежує nucleus-sampling для стабільності |
| `top_k` | **20** | Не дозволяє малоймовірним токенам проникати у вивід |
| `max_output_tokens` | **2048** | Достатньо для будь-якого запиту + JSON-обгортки |

## Обробка помилок

| Виняток | Причина | Обробка |
|---------|---------|---------|
| `ResourceExhausted` | Перевищено ліміт Free Tier | Зрозуміле повідомлення + підказка зачекати |
| `InvalidArgument` | Невірний ключ або формат | Описова помилка з деталями |
| `ValueError` | Модель повернула не-JSON або невідповідну схему | Повне повідомлення про помилку + сирий вивід для налагодження |

## Кешування

Успішні результати зберігаються у `.query_cache/` (TTL = 1 год.).  
Ключ кешу — SHA-256 від `(нормалізований_запит, md5_хеш_схеми)`.  
Кешування вмикається/вимикається перемикачем у боковій панелі.
